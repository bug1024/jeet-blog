# Bug1024 博客文章写作与摄影规范

## 1. 内容定位

博客用于建立 Bug1024 的长期个人品牌，主要记录：

- 工作中的系统、协作、组织与人的判断；
- 生活中的选择、时间、家庭与个人变化；
- 少量来自真实实践、值得长期保留的技术内容。

文章从具体经历或问题出发。表达应具体、克制、有判断，不追逐热点，不把尚未消化的观点包装成答案，也不为了显得深刻而制造深刻。

默认是单向对外输出。不要在结尾机械地邀请读者留言、联系、讨论或产生共鸣；只有文章本身确实需要提问时，才保留问题。

## 2. 文案语气

### 应该

- 先写事实和经历，再给出判断；
- 区分事实、推测和个人感受；
- 使用自然、准确的现代汉语；
- 保留暂时没有结论的问题；
- 标题清楚表达文章讨论的对象。

### 避免

- 空泛格言和刻意提炼的“金句”；
- “保持热爱”“寻找答案”“与时间和解”等泛化表达；
- 把普通经历包装成宏大叙事；
- 没有内容作用的文学修饰；
- 默认使用“欢迎交流”“如果你也……”“希望与你产生共鸣”等社交型结尾；
- 为凑更新频率而写作。

文学感应来自具体观察、准确节奏和真实细节；文化感应来自视野和判断，而不是古典词语或故作深沉。

## 3. 文章结构

结构服务于内容，不强制所有文章套用相同提纲。通常可以包括：

1. 一件具体的事、一个变化或一个问题；
2. 对其背景和机制的解释；
3. 自己形成或修正的判断；
4. 当前结论、行动或仍未解决的部分。

长文章使用二级标题帮助阅读；短文章不为了形式完整强行拆章节。

## 4. Front matter

基础字段：

```yaml
---
title: "文章标题"
description: "用一两句话说明文章具体讨论什么，不写宣传语。"
date: 2026-07-18
author: "Bug1024"
slug: "stable-english-slug"
draft: true
tags:
  - 工作与系统
---
```

规则：

- 文件名：`YYYY-MM-DD-topic.md`；
- `slug`：简短、稳定、无日期，发布后不随意修改；
- `description`：承担文章导语和列表摘要作用，应包含有效信息；
- `tags`：优先复用已有标签，不为细小差异不断创建同义标签；
- 用户未明确要求发布时保留 `draft: true`。

可选视觉关联字段：

```yaml
visual: "bug-visual-001"
```

该字段只用于关联“第 1024 格”视觉作品档案。视觉隐喻与 Prompt 存放在 `data/visuals/<id>.yaml`，不得写入 `alt`。

## 5. 文章标题区的三种模式

### 无摄影题图

默认使用钢蓝文章坐标板。坐标板展示真实的文章序号、日期和阅读时间，不需要在 front matter 中配置。

禁止为了避免无图而添加与正文无关的网络图片、生成图或旧站装饰图。

### “第 1024 格”生成封面

只有图像能够具体解释文章，而不是单纯填补空白时才使用。生成和接入过程必须遵循 `skills/blog-cover-generator/SKILL.md`。

每张正式采用的 AI 生成封面必须：

- 使用连续的 `bug-visual-NNN` 编号；
- 在 `data/visuals/` 中建立独立档案；
- 在文章顶层 front matter 中通过 `visual` 关联档案；
- 保留客观 `alt`，将自然秩序、Bug、钢蓝作用、隐喻和最终 Prompt 写入档案；
- 与图片文件、文章配置一起纳入 Git 管理。

### 有摄影题图

只有当照片与文章存在真实关系时使用：

- 拍摄于文章发生的地点或时期；
- 与文章讨论的空间、时间或具体观察有关；
- 照片促成了文章；
- 照片补充了文字无法表达的信息。

摄影图不是通用封面装修。照片来自作者本人时，优先保留真实的地点、时间和构图意图。

预留的 front matter 合约：

```yaml
cover:
  image: "/images/article/example.jpg"
  alt: "客观说明照片中可见的内容"
  caption: "南京 · 2026 年 4 月"
  position: "50% 40%"
  fit: "cover"
```

字段含义：

- `image`：站内图片路径；
- `alt`：无障碍与搜索使用的客观画面描述；
- `caption`：地点、时间或一句具体观察；
- `position`：裁切焦点，默认 `center`；
- `fit`：默认 `cover`，必须完整保留构图时使用 `contain`。

如果当前模板尚未支持某个字段，Codex 应先说明并补齐模板支持，不能悄悄忽略作者配置。

## 6. 摄影题图尺寸

### 标准画框

| 项目 | 规范 |
| --- | --- |
| 比例 | `4:3` |
| 推荐原图 | `2400 × 1800px` |
| 最低原图 | `1600 × 1200px` |
| 桌面展示 | 约 `328 × 246px` |
| 移动展示 | 内容容器宽度 `100%`，保持 `4:3` |
| 色彩空间 | `sRGB` |
| 默认适配 | `object-fit: cover` |
| 完整构图 | `object-fit: contain` + 低对比度雾灰画框 |

统一的是画框、基线和页面节奏，不是强迫每张照片采用同一种裁切。

### 图片输出

推荐由 Hugo 生成响应式宽度：

- `640w`
- `960w`
- `1440w`
- `1920w`

优先输出 WebP，并保留兼容格式。建议体积：

- 桌面加载版本 `150–300KB`；
- 移动加载版本 `80–160KB`。

不得直接让网页加载相机原始大图。

### 横图、方图与竖图

- 横图：默认进入 `4:3` 页头画框；
- 方图：根据边缘是否重要选择轻微裁切或 `contain`；
- 竖图：默认进入正文，不强行裁成页头横图。必要时另行设计竖图布局。

## 7. 构图与裁切

默认使用 `fit: cover`，但必须检查裁切结果。照片主体不在中心时，通过 `position` 指定焦点，例如：

```yaml
position: "30% 50%"
position: "70% 35%"
```

如果边缘本身属于构图，不得为了填满画框而破坏照片；改用：

```yaml
fit: "contain"
```

Codex 在处理作者照片时不得擅自：

- 改变照片内容或生成不存在的物体；
- 使用强滤镜、渐变、投影或装饰蒙版；
- 为适配页面而忽略原始构图；
- 把照片当作文字背景并在上面叠加标题。

## 8. Alt 与图注

`alt` 与 `caption` 承担不同职责：

- `alt` 客观说明画面里有什么；
- `caption` 记录地点、时间或拍摄时的具体观察。

推荐图注：

```text
南京 · 2026 年 4 月
```

```text
下午五点，建筑的影子越过窗沿。
```

避免图注：

```text
生活处处是风景。
在光影中寻找答案。
时间留下的痕迹。
```

## 9. 正文图片

- 图片应与上下文存在明确关系；
- 图注紧随图片，不用图片文件名代替说明；
- 不连续堆放多张近似照片；
- 竖图保持合理最大宽度，不放大到模糊；
- 图片可以突破正文窄栏，但不得超出全站内容网格；
- 技术截图优先保证文字可读，不套摄影画框。

## 10. 发布前检查

### 内容

- 标题与正文讨论的是同一件事；
- `description` 有信息量，不是宣传语；
- 结论有事实或经历支撑；
- 删除可有可无的金句、升华和社交邀请；
- 标签复用了现有体系。

### 摄影

- 照片与文章确实相关；
- 已检查桌面和移动裁切；
- `alt`、`caption`、`position`、`fit` 正确；
- 图片尺寸和体积合适；
- 竖图没有被强行裁成横图。
